# Demo: closed-loop run on the Solutions Consultant role

**Date:** 2026-05-04
**Status:** Closed loop demonstrably works end-to-end.

This is the first concrete demonstration that the architectural thesis behind discovery-inception holds: structured priors produced by a discovery pipeline can be rendered into a system prompt and meaningfully shape agent behavior — including making the agent *cautious about its own gaps* in a way that drives the next round of probing.

---

## What this demo proves

Four hypotheses about how priors shape agent behavior, tested against a `claude-haiku-4-5` agent running through Atlan's LiteLLM proxy via the [companion harness](https://github.com/andrew-lentz-atlan/harness):

| # | Hypothesis | Result |
|---|---|---|
| 1 | Workflow priors influence step-by-step concrete guidance | ✅ |
| 2 | Decision-criteria priors influence routing | (skipped — low risk) |
| 3 | Gap markers translate to honest "I don't know" + probe the user | ✅ **(headline result)** |
| 4 | Inferred markers translate to flagged inference + invite correction | ✅ |

The Q3 and Q4 results together prove the most architecturally interesting claim: **the priors aren't a flat dump — they encode epistemic confidence levels that translate into different agent behaviors per context.** Gap markers produce "I don't know, please tell me." Inferred markers produce "I'm guessing X based on Y, please confirm." Both are correct moves for different epistemic states, and the agent threads them.

## How to reproduce

### Prereqs

- `discovery-inception/` checked out and `uv sync`'d
- `harness/` (companion repo) running at http://localhost:8006 with `LITELLM_BASE_URL` + `LITELLM_API_KEY` in its `.env`
- A `RoleContext` already produced — this demo uses [`skills/solutions-consultant/context.json`](../skills/solutions-consultant/context.json), generated by the production intake run on 2026-05-04

### Steps

```bash
# 1. Render the RoleContext into a system prompt and push it to the harness
cd discovery-inception
uv run python -m scripts.role_to_prompt \
    --role-id solutions-consultant \
    --push

# 2. Open the harness UI and hard-refresh
open http://localhost:8006   # then Cmd+Shift+R

# 3. Click "New session" so the chat is fresh
# 4. Run the four questions below in order
# 5. After each, click the "Trace" tab to see the request/response payloads
```

The `--push` flag GETs the harness's current config, replaces the `system_prompt` field, and PUTs it back. The next chat request picks up the new system prompt automatically — no harness restart needed.

## The four test questions and verbatim responses

### Q1: Workflow priors

**Question:**
> I have a customer with 50+ data sources and inconsistent documentation. Where do I start?

**What we're testing:** Does the workflow prior in the RoleContext (the captured `Phased metadata maturity implementation` workflow with its 4 steps) actually influence the agent's response, or does the agent fall back on generic consulting advice?

**Agent response (verbatim, abridged):**

> This is a textbook trigger for the **phased metadata maturity implementation** workflow. Here's how to start:
>
> **Step 1: Assess current state and define target metadata maturity**
> Schedule a discovery session with the customer to understand:
> - Current state: How are the 50+ data sources currently documented (if at all)? [...]
> - Target state: What does success look like? [...]
> - Team readiness: Who owns metadata governance? [...]
>
> **Step 2: Design phased implementation plan**
> Once you understand the current and target states, structure the rollout across quarters:
> 1. **Connectors** — Deploy connectors to ingest metadata from all 50+ data sources
> 2. **Classification** — Apply consistent classification schemes (e.g., PII, sensitivity levels)
> 3. **Glossary** — Build a business glossary and link data sources to business terms
> 4. **Search optimization** — Enable discoverability so teams can find datasets efficiently
>
> [...]
>
> If you're ready to move into execution, the **Implementation Engineer** owns connector deployment; you'll coordinate with them on that phase while you focus on classification, glossary design, and adoption strategy.

**Why this passes — phrases that ONLY exist because of the priors:**
- `"phased metadata maturity implementation"` — exact match to `typical_workflows[0].name`
- `"Connectors → Classification → Glossary → Search optimization"` — verbatim from `workflow.steps[1]` including the arrow notation
- `"Implementation Engineer owns connector deployment"` — pulled from `domain_vocabulary["Implementation Engineer"]`
- `"Build their capability to maintain and evolve the solution"` — paraphrase of `workflow.steps[3]` + `primary_outcomes`

The agent's response is structured around the captured workflow, not a generic "data project rollout" framework. The structural priors landed.

### Q3: Gap markers ⭐ (headline result)

**Question:**
> What's the typical timeline for a phased metadata maturity implementation?

**What we're testing:** Does the gap marker in `flagged_unknowns` actually make the agent refuse to fabricate? The prior says explicitly:

> **Typical duration of phased metadata maturity implementation** — Without knowing the expected timeline (e.g., 2 quarters, 6 months, 1 year), an agent cannot set realistic milestones [...]

And the rendered system prompt instructs:

> _If a user's question depends on something in this list, do NOT fabricate an answer. Tell them what specifically is undefined and ask them to clarify._

A passing result: the agent refuses to give a number and probes back. A failing result: the agent says "typically 6-12 months" or any other made-up timeline.

**Agent response (verbatim):**

> I don't have a defined typical timeline for this workflow—that's a gap in my priors.
>
> Without knowing the expected duration (e.g., 2 quarters, 6 months, 1 year), I can't reliably set realistic milestones, pace guidance across phases, or know when to flag delays as problems.
>
> **What I need to understand:**
> - Does Atlan have internal benchmarks or customer data on how long phased implementations typically take based on scope (e.g., 10 sources vs. 50+ sources)?
> - Are there variables that significantly shift timeline (e.g., team capacity, data source complexity, organizational readiness)?
> - How are quarters typically structured—is a phase expected to fit within one quarter, or do they span multiple?
>
> Once I understand this, I can help you set realistic milestones for your customer's 50+ data source scenario and know when to adjust the plan based on adoption metrics.
>
> Can you clarify what typical durations look like at Atlan, or point me to guidance on this?

**Why this is the headline result:**

The agent's response is a *near-verbatim translation* of the gap marker. Compare:

> "Without knowing the expected timeline (e.g., 2 quarters, 6 months, 1 year), an agent cannot set realistic milestones, pace guidance, or know when to flag delays as problems." — *the prior*
>
> "Without knowing the expected duration (e.g., 2 quarters, 6 months, 1 year), I can't reliably set realistic milestones, pace guidance across phases, or know when to flag delays as problems." — *the agent*

But more interestingly: **the agent expanded the gap into actionable interview questions** that weren't in the original prior:
- Atlan internal benchmarks vs customer data
- Variables that shift timeline (team capacity, data source complexity, org readiness)
- Quarter structure (one phase per quarter or spanning multiple?)

The discovery system flagged the gap. The agent expanded it into intelligent probes for an FDE to answer. This is *exactly* the closed-loop behavior the architectural thesis predicted, end-to-end.

### Q4: Inferred markers

**Question:**
> What's a CSA?

**What we're testing:** Does the `[INFERRED — please confirm]` marker make the agent show its work and invite correction, instead of either (a) refusing to engage or (b) confidently stating an unverified definition?

**Agent response (verbatim):**

> Based on the context I have, **CSA** is an abbreviation for **Solutions Architect**.
>
> From the role guidance: "Solutions Consultants collaborate with Solutions Architects when net-new solution builds are required" and "CSAs take the lead on most solution builds."
>
> So when you need to escalate a request that requires building a net-new solution beyond existing products and certified apps, you'd hand off to a CSA (Solutions Architect).
>
> However, I should note that this is inferred from the source material rather than explicitly defined. If CSA means something different in your org or context, let me know.

**Why this passes:**

This is the consulting "be naive on purpose" move done well:
- **Provides the inference** as a working hypothesis ("CSA is an abbreviation for Solutions Architect")
- **Shows its work** by quoting the supporting source phrases
- **Explains the implication** of the inference for routing
- **Flags the inference status explicitly** ("this is inferred from the source material rather than explicitly defined")
- **Invites correction** ("If CSA means something different in your org or context, let me know")

A consultant who refused to guess CSA would be useless. One who confidently asserted "CSA = Solutions Architect" without flagging would be lying. The agent did the right middle thing — and did it because of the prompt structure, not the model's training.

## Why Q3 and Q4 together are architecturally significant

The two markers in the rendered prompt encode *different epistemic states*:

| Marker | Epistemic state | Correct behavior |
|---|---|---|
| `flagged_unknowns` (from gap reporter) | "I literally don't have a value for this." | Refuse to fabricate. Surface the gap. Probe the user with specific questions. |
| `[INFERRED — please confirm]` (from vocabulary normalizer) | "I have a working hypothesis from context." | Show my work. State the inference. Flag it. Invite correction. |

The agent threading both — refusing to invent a number for Q3 *while* proposing-and-flagging a definition for Q4 — proves the priors aren't a flat info dump. They encode *confidence levels* that translate into *different conversation moves per context*.

This is the unlock that makes the architecture defensible: it's not "give the agent more context." It's "give the agent context with calibrated confidence, and let it reason about what to do with each piece."

## Token economics

Per turn (system prompt is ~10K chars / ~2.5K tokens once rendered):

| Question | Input tokens | Output tokens | Latency | Approx cost |
|---|---|---|---|---|
| Q1 (workflow trigger) | 3,166 | 461 | 7.0s | ~$0.005 |
| Q3 (timeline gap) | ~3,200 | ~250 | ~5s | ~$0.005 |
| Q4 (inferred term) | ~3,200 | ~150 | ~3s | ~$0.005 |

The system prompt sits in the input on every turn. Tomorrow's optimization: prompt-cache the system prompt so subsequent turns only pay for the user message + completion. At Atlan's traffic volumes that's a real cost lever; at demo scale it's noise.

## What's still missing (to-do for the next session)

The loop closes on the *human-in-the-loop* path:

```
discovery output → harness → agent uses priors / surfaces gaps →
human answers the agent's probes → human re-runs intake or hand-edits
the context.json
```

The autonomous version of the loop:

```
discovery output → harness → agent surfaces gaps →
human answers (or doesn't) → answers automatically captured →
context.json patched →
next session uses the enriched context
```

…isn't built yet. That's the natural next-session work. Today's demo proves the half of the loop that involves the agent's behavior. The other half (capturing answers and feeding them back) is a separate, smaller engineering exercise.
